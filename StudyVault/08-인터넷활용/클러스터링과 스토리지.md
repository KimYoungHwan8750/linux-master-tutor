## 1. RAID (Redundant Array of Independent Disks)

> [!info] RAID는 여러 개의 물리 디스크를 결합하여 **성능 향상**, **데이터 안정성**, 또는 둘 다를 제공하는 기술이다.

### RAID 레벨 비교표

| 구분 | RAID 0 | RAID 1 | RAID 5 | RAID 6 | RAID 1+0 |
|------|--------|--------|--------|--------|----------|
| 이름 | **스트라이핑** | **미러링** | **스트라이핑+패리티** | 이중 패리티 | 미러+스트라이핑 |
| 최소 디스크 | **2개** | **2개** | **3개** | 4개 | 4개 |
| 용량 효율 | 100% | **50%** (2배 필요) | (n-1)/n | (n-2)/n | 50% |
| 읽기 속도 | 빠름 | 빠름 | 빠름 | 빠름 | 빠름 |
| 쓰기 속도 | **빠름** | 보통 | 보통 | 느림 | 보통 |
| 장애 허용 | **불가** | 1개 | **1개** | 2개 | 1개/그룹 |
| 복구 가능 | **X** | O | O | O | O |
| 용도 | 임시 데이터, 속도 | 중요 데이터 백업 | **실무 다용** | 높은 안정성 | 성능+안정성 |

### 각 RAID 상세

#### RAID 0 (스트라이핑, Striping)

```
       데이터: A B C D E F G H

  디스크1    디스크2
  ┌──────┐  ┌──────┐
  │  A   │  │  B   │
  │  C   │  │  D   │
  │  E   │  │  F   │
  │  G   │  │  H   │
  └──────┘  └──────┘
  ← 분산 저장 (속도↑) →
```

- 데이터를 **분산 저장** (스트라이핑)
- 최소 **2개** 디스크 필요
- **속도 향상**: 동시 읽기/쓰기
- **복구 불가**: 하나의 디스크 고장 → 전체 데이터 손실
- 용량: 디스크 수 x 단일 용량

#### RAID 1 (미러링, Mirroring)

```
       데이터: A B C D

  디스크1    디스크2
  ┌──────┐  ┌──────┐
  │  A   │  │  A   │  ← 동일 복제
  │  B   │  │  B   │
  │  C   │  │  C   │
  │  D   │  │  D   │
  └──────┘  └──────┘
  ← 미러링 (안정성↑) →
```

- **동일한 내용**을 2개 이상의 디스크에 **백업**
- 실제 용량의 **2배** 필요
- **신뢰성** 높음: 하나의 디스크 고장 → 다른 디스크로 복구 가능
- 읽기 속도 향상, 쓰기 속도 보통

#### RAID 5 (스트라이핑 + 패리티)

```
       데이터: A B C + 패리티(P)

  디스크1    디스크2    디스크3
  ┌──────┐  ┌──────┐  ┌──────┐
  │  A1  │  │  A2  │  │  Ap  │  ← 패리티 분산
  │  B1  │  │  Bp  │  │  B2  │
  │  Cp  │  │  C1  │  │  C2  │
  └──────┘  └──────┘  └──────┘
  ← 패리티가 각 디스크에 분산 →
```

- **스트라이핑** + **패리티 (오류 복구 정보)** 조합
- 최소 **3개** 디스크 필요
- **실무에서 가장 많이 사용**
- 1개 디스크 고장 시 패리티로 **복구 가능**
- 용량: (n-1) x 단일 용량

## 2. LVM (Logical Volume Manager)

> [!info] LVM은 물리 디스크의 저장 공간을 **논리적으로 유연하게 관리**할 수 있는 시스템이다.

### LVM 구조

```
┌─────────────────────────────────────────────────────┐
│                    Logical Volumes (LV)              │
│        ┌──────────┐  ┌──────────┐  ┌──────────┐     │
│        │  LV: /   │  │ LV: /home│  │ LV: /var │     │
│        │  20GB    │  │  50GB    │  │  30GB    │     │
│        └────┬─────┘  └────┬─────┘  └────┬─────┘     │
│             └──────────────┼──────────────┘           │
│  ┌─────────────────────────┴─────────────────────┐   │
│  │            Volume Group (VG)                   │   │
│  │            vg_data (100GB)                     │   │
│  └───────┬──────────────┬──────────────┬─────────┘   │
│     ┌────┴─────┐  ┌────┴─────┐  ┌────┴─────┐       │
│     │  PV      │  │  PV      │  │  PV      │       │
│     │ /dev/sda │  │ /dev/sdb │  │ /dev/sdc │       │
│     │  40GB    │  │  30GB    │  │  30GB    │       │
│     └──────────┘  └──────────┘  └──────────┘       │
│          Physical Volumes (PV) - 물리 볼륨           │
│                                                     │
│     ┌──────────┐  ┌──────────┐  ┌──────────┐       │
│     │ 디스크1   │  │ 디스크2   │  │ 디스크3   │     │
│     └──────────┘  └──────────┘  └──────────┘       │
│          Physical Disks - 물리 디스크                │
└─────────────────────────────────────────────────────┘
```

### LVM 구성 요소

| 구성 | 약자 | 설명 |
|------|------|------|
| **Physical Volume** | **PV** | 물리 볼륨 — 실제 디스크 파티션 |
| **Volume Group** | **VG** | 볼륨 그룹 — PV들을 하나로 묶은 풀(Pool) |
| **Logical Volume** | **LV** | 논리 볼륨 — VG에서 할당받은 가상 파티션 (실제 사용) |
| **Physical Extent** | **PE** | 물리 확장 — PV를 구성하는 기본 블록 단위 |

### LVM 명령어

| 작업 | PV | VG | LV |
|------|-----|-----|-----|
| **생성** | `pvcreate` | `vgcreate` | `lvcreate` |
| **확인** | `pvdisplay` | `vgdisplay` | `lvdisplay` |
| **목록** | `pvscan` / `pvs` | `vgscan` / `vgs` | `lvscan` / `lvs` |
| **삭제** | `pvremove` | `vgremove` | `lvremove` |
| **확장** | — | `vgextend` | `lvextend` |
| **축소** | — | `vgreduce` | `lvreduce` |

```bash
# LVM 생성 순서 예시
# 1. PV 생성
pvcreate /dev/sdb1 /dev/sdc1

# 2. VG 생성 (PV를 묶어서)
vgcreate vg_data /dev/sdb1 /dev/sdc1

# 3. LV 생성 (VG에서 할당)
lvcreate -L 20G -n lv_home vg_data

# 4. 파일시스템 생성
mkfs.ext4 /dev/vg_data/lv_home

# 5. 마운트
mount /dev/vg_data/lv_home /home

# 확인
pvdisplay    # PV 상태 확인
vgdisplay    # VG 상태 확인
lvdisplay    # LV 상태 확인

# LV 확장
lvextend -L +10G /dev/vg_data/lv_home
resize2fs /dev/vg_data/lv_home
```

> [!important] LVM 핵심 순서
> **PV → VG → LV** (물리 → 그룹 → 논리)
> - `pvcreate` → `vgcreate` → `lvcreate`

## 3. 클러스터링 (Clustering)

> [!info] 클러스터링은 여러 대의 컴퓨터를 하나의 시스템처럼 묶어서 사용하는 기술이다.

### 클러스터 유형 비교표

| 유형 | 풀네임 | 목적 | 특징 | 키워드 |
|------|--------|------|------|--------|
| **HPC** | High Performance Computing | **고성능 계산** | 대규모 연산 작업 | **베어울프(Beowulf) 클러스터** |
| **LVS** | Linux Virtual Server | **부하 분산** | 트래픽 분산 처리 | **로드밸런서** |
| **HA** | High Availability | **고가용성** | 서비스 중단 최소화 | **IP 이주 (Failover)** |

### 각 클러스터 상세

#### HPC (High Performance Computing)

- **고성능 계산** 클러스터
- **베어울프(Beowulf)** 클러스터: 리눅스 기반 고성능 계산 클러스터의 대표적 형태
- 과학 시뮬레이션, 기상 예측, 유전체 분석 등에 사용
- 저가 하드웨어를 묶어 슈퍼컴퓨터급 성능 달성

#### LVS (Linux Virtual Server)

- **부하 분산(Load Balancing)** 클러스터
- 외부 요청을 여러 서버에 **분산** 처리
- **로드밸런서**가 트래픽을 분배
- 웹 서비스, 대규모 트래픽 처리에 사용

#### HA (High Availability)

- **고가용성** 클러스터
- 장애 발생 시 서비스를 **다른 서버로 자동 이전**
- **IP 이주(Failover)**: 장애 서버의 IP를 대기 서버로 자동 이전
- 서비스 중단 시간 최소화 목표

## 4. 서버 가상화

> [!info] 가상화는 하나의 물리 서버에서 여러 가상 머신(VM)을 실행하는 기술이다.

### 가상화 솔루션 비교표

| 솔루션 | 가상화 방식 | 특징 | 키워드 |
|--------|------------|------|--------|
| **Xen** | **반가상화** (Paravirtualization) | CPU 반가상화, 높은 성능 | **반가상화** |
| **KVM** | **전가상화** (Full Virtualization) | CPU 전가상화, 레드햇 하이퍼바이저 | **레드햇**, 커널 기반 |
| **VirtualBox** | 전가상화 | 독자적 가상 장치 관리자, 데스크톱용 | **오라클**, GUI |
| **Docker** | **컨테이너** 기반 | OS 수준 가상화, 경량 | **컨테이너**, 이미지 |

### 전가상화 vs 반가상화

| 구분 | 전가상화 (Full) | 반가상화 (Para) |
|------|----------------|-----------------|
| 게스트 OS 수정 | 필요 없음 | **필요** (커널 수정) |
| 하드웨어 에뮬레이션 | 완전 에뮬레이션 | 부분 에뮬레이션 |
| 성능 | 오버헤드 있음 | **성능 우수** |
| 호환성 | 높음 | 낮음 (수정 필요) |
| 대표 솔루션 | **KVM**, VirtualBox | **Xen** |

### 하이퍼바이저 유형

| 유형 | 설명 | 예시 |
|------|------|------|
| **Type 1** (베어메탈) | 하드웨어 위에 직접 설치 | Xen, KVM, VMware ESXi |
| **Type 2** (호스트형) | OS 위에 설치 | VirtualBox, VMware Workstation |

```
Type 1 (베어메탈)          Type 2 (호스트형)
┌──────┐ ┌──────┐         ┌──────┐ ┌──────┐
│ VM 1 │ │ VM 2 │         │ VM 1 │ │ VM 2 │
└──┬───┘ └──┬───┘         └──┬───┘ └──┬───┘
┌──┴────────┴───┐         ┌──┴────────┴───┐
│ 하이퍼바이저   │         │ 하이퍼바이저   │
└──────┬────────┘         └──────┬────────┘
┌──────┴────────┐         ┌──────┴────────┐
│   하드웨어     │         │   호스트 OS    │
└───────────────┘         └──────┬────────┘
                          ┌──────┴────────┐
                          │   하드웨어     │
                          └───────────────┘
```

### Docker (컨테이너 기반 가상화)

- **컨테이너** 기반 가상화 플랫폼
- OS 수준의 가상화 (하이퍼바이저 불필요)
- **이미지(Image)** 기반 배포
- 장점: **경량**, 빠른 시작, 이식성
- 전통적 VM과 달리 **게스트 OS 불필요**

## 5. 클라우드 컴퓨팅 (Cloud Computing)

> [!info] 클라우드 컴퓨팅은 인터넷을 통해 컴퓨팅 자원을 필요한 만큼 **서비스 형태**로 제공하는 모델이다.

### 서비스 모델 비교표

| 모델 | 풀네임 | 제공 범위 | 설명 | 대표 예시 |
|------|--------|-----------|------|-----------|
| **IaaS** | Infrastructure as a Service | **하드웨어** 임대 | 서버, 스토리지, 네트워크 | **Amazon EC2**, Azure VM |
| **PaaS** | Platform as a Service | **개발 플랫폼** 제공 | OS, 런타임, 개발도구 | **Google App Engine**, Heroku |
| **SaaS** | Software as a Service | **응용프로그램** 서비스 | 완성된 소프트웨어 | **Gmail**, **네이버 클라우드**, Office 365 |

```
┌─────────────────────────────────────────────┐
│           관리 영역 비교                      │
│                                             │
│  온프레미스    IaaS       PaaS      SaaS    │
│  ┌──────┐  ┌──────┐  ┌──────┐  ┌──────┐   │
│  │ 앱   │  │ 앱   │  │ 앱   │  │██████│   │
│  │ 데이터│  │ 데이터│  │ 데이터│  │██████│   │
│  │ 런타임│  │ 런타임│  │██████│  │██████│   │
│  │ 미들웨어│ │ 미들웨어│ │██████│  │██████│   │
│  │ OS   │  │ OS   │  │██████│  │██████│   │
│  │ 가상화│  │██████│  │██████│  │██████│   │
│  │ 서버 │  │██████│  │██████│  │██████│   │
│  │ 스토리지│ │██████│  │██████│  │██████│   │
│  │ 네트워크│ │██████│  │██████│  │██████│   │
│  └──────┘  └──────┘  └──────┘  └──────┘   │
│                                             │
│  ███ = 클라우드 공급자 관리                   │
│  빈칸 = 사용자 관리                          │
└─────────────────────────────────────────────┘
```

> [!tip] 클라우드 서비스 기억법
> - **I**aaS = **I**nfrastructure (인프라/하드웨어) = Amazon EC2
> - **P**aaS = **P**latform (플랫폼/개발환경) = Google App Engine
> - **S**aaS = **S**oftware (소프트웨어/완성 앱) = Gmail

## 6. 하둡 (Hadoop)

- **대용량 데이터** 분산 처리 프레임워크
- **자바(Java)** 기반 오픈소스
- Apache 재단 프로젝트
- 핵심 구성:
  - **HDFS** (Hadoop Distributed File System): 분산 파일 시스템
  - **MapReduce**: 분산 병렬 처리 프레임워크
- 대량의 데이터를 여러 서버에 분산 저장하고 병렬 처리

## 시험 빈출 패턴

| 시나리오/키워드 | 정답 |
|----------------|------|
| 분산 저장, 속도↑, 복구 불가 | **RAID 0** (스트라이핑) |
| 동일 백업, 2배 용량 | **RAID 1** (미러링) |
| 스트라이핑+패리티, 최소 3개 | **RAID 5** |
| 실무에서 가장 많이 쓰는 RAID | **RAID 5** |
| LVM 구조 순서 | **PV → VG → LV** |
| PV 생성 명령어 | `pvcreate` |
| VG 생성 명령어 | `vgcreate` |
| LV 생성 명령어 | `lvcreate` |
| PV 상태 확인 | `pvdisplay` |
| VG 상태 확인 | `vgdisplay` |
| LV 상태 확인 | `lvdisplay` |
| 고성능 계산 클러스터 | **HPC** (베어울프) |
| 부하 분산 클러스터 | **LVS** (로드밸런서) |
| 고가용성 클러스터 | **HA** (IP 이주) |
| CPU 반가상화 | **Xen** |
| CPU 전가상화, 레드햇 | **KVM** |
| 컨테이너 기반 가상화 | **Docker** |
